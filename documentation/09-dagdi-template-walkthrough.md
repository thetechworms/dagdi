# 09 - `dagdi-template.yaml` Walkthrough

This file explains every section of the template generated by `dagdi config generate` (written to `~/.config/dagdi/dagdi-template.yaml` by default).

## Template Intent

The template demonstrates:

- One product (`myapp`)
- Three environments (`dev`, `staging`, `prod`)
- Per-environment server topology
- Mixed service types (`systemd`, `docker`)
- Optional global settings

## Top-Level Layout

```yaml
products:
  - name: myapp
    environments:
      - name: dev
      - name: staging
      - name: prod

global_settings:
  ssh_timeout: 30
  on_partial_failure: prompt
  live_status_table: false
  theme: default
```

## Environment Structure

Each environment includes `servers`, and each server contains:

- `name`
- `type`
- `ips`
- `ssh`
- `services`

## `dev` Example

`dev` shows a minimal two-tier setup:

- `web-1`
  - services: `nginx` (systemd), `api` (docker)
- `db-1`
  - service: `postgres` (systemd)

## `staging` Example

`staging` mirrors production-like roles with staging-specific names:

- `web-staging`
- `db-staging`

This gives safe command practice scope before production operations.

## `prod` Example

`prod` demonstrates:

- Multiple web servers (`web-prod-1`, `web-prod-2`)
- Multiple IPs per server
- Shared service names across hosts (`nginx`, `api`)
- Dedicated DB host (`db-prod`)

Operational implication:

- Service/metrics actions run per `(server, ip)` target.
- For multi-IP servers, actions run once per IP.

## SSH Block in Template

Templates can define SSH either per server or once at the environment level under `servers.ssh`. Shared environment-level SSH is useful when most machines use the same login details and only a few need overrides.

Template uses key-based auth:

```yaml
ssh:
  username: ubuntu
  key_path: ~/.ssh/id_rsa
  timeout: 30
```

Shared SSH example:

```yaml
servers:
  ssh:
    username: ubuntu
    key_path: ~/.ssh/id_rsa
    timeout: 30
    sudo_password: prompt
  items:
    - name: web-1
      ssh:
        timeout: 60
```

Optional keys not always shown in every server block:

- `password`
- `sudo`
- `sudo_password`

Use `sudo_password: prompt` to avoid storing sudo passwords in plain text. Put it
under a server's `ssh` to prompt once for that server, or under shared `servers.ssh`
to prompt once and reuse it across all servers in that environment.

## Service Blocks in Template

### systemd examples

```yaml
- name: nginx
  type: systemd

- name: postgres
  type: systemd
```

### docker examples

```yaml
- name: api
  type: docker
  config:
    container_name: myapp-api
```

Why `container_name` is helpful:

- Avoids dependency on service name matching container runtime name.

## Global Settings in Template

```yaml
global_settings:
  ssh_timeout: 30
  on_partial_failure: prompt
  live_status_table: false
  theme: default
```

Semantics:

- `ssh_timeout`: intended default timeout
- `on_partial_failure`: intended failure policy
- `live_status_table`: feature flag for progressive status-table updates while checks run
- `theme`: color theme for CLI output (`default`, `light`, `dark`, `no_color`)

Implementation note:

- Values are loaded/validated and visible in `show-settings`; runtime enforcement is partial in current code.
- `theme` is fully enforced at runtime: all output colors and styles are driven by the selected theme.

## Adapting Template to Real Infra

1. Rename `myapp` to your real product name.
2. Keep environment names stable (`dev/staging/prod` or your own conventions).
3. Replace all private IP placeholders with real target addresses.
4. Update SSH usernames and keys per environment.
5. Keep service names consistent with your real process/container/service manager names.
6. Validate after every edit:

```bash
dagdi config validate
```

## Recommended File Strategy

- Keep one product in one file to satisfy merge constraints.
- For many products, create separate files like:
  - `~/.config/dagdi/dagdi-payments.yaml`
  - `~/.config/dagdi/dagdi-analytics.yaml`
